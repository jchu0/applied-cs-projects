# Distributed Job Queue - API Documentation

This document describes the REST API exposed by the FastAPI application in
`src/jobqueue/api.py`. Every endpoint below is implemented by that app; there
are no other public routes.

## Base URL

```
http://localhost:8000
```

The API server runs on port 8000 by default (see `__main__` in `api.py`).
Interactive documentation is generated by FastAPI and available at:

- Swagger UI: `http://localhost:8000/docs`
- ReDoc: `http://localhost:8000/redoc`
- OpenAPI schema: `http://localhost:8000/openapi.json`

## Authentication

Authentication is **opt-in** and implemented by `src/jobqueue/security.py`
(`install_hardening`). It is controlled entirely by environment variables and is
disabled by default so the quick-start works out of the box.

- `API_KEYS` — comma-separated list of valid keys. Unset/empty **disables auth**.
- `RATE_LIMIT_PER_MINUTE` — integer, default `120`. `0` disables rate limiting.
- `REQUEST_TIMEOUT_SECONDS` — float, default `30`. `0` disables the timeout.

When `API_KEYS` is set, every request must present a valid key via either header:

```http
Authorization: Bearer <api_key>
```

or

```http
X-API-Key: <api_key>
```

The following paths are always open (never require a key and are never rate
limited): `/`, `/health`, `/ready`, `/readiness`, `/metrics`, `/docs`,
`/redoc`, `/openapi.json`, and their sub-paths. **All other routes are
protected**, including the worker-facing `/internal/*` routes.

Auth/limit/timeout failures return a JSON body of the form
`{"detail": "..."}` with these status codes:

- `401` — missing or invalid API key (adds `WWW-Authenticate: Bearer`).
- `429` — rate limit exceeded (adds a `Retry-After` header).
- `504` — request exceeded `REQUEST_TIMEOUT_SECONDS`.

## Quick Reference

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/tasks` | POST | Create and enqueue a task |
| `/tasks/{task_id}` | GET | Get a task by ID |
| `/tasks/{task_id}` | DELETE | Cancel a pending task |
| `/tasks/{task_id}/result` | GET | Get the result of a completed task |
| `/tasks/{task_id}/retry` | POST | Requeue a failed/cancelled task |
| `/queues` | GET | List all known queues |
| `/queues/{queue_name}/stats` | GET | Get statistics for a queue |
| `/internal/dequeue` | POST | (Worker) Dequeue a task |
| `/internal/acknowledge` | POST | (Worker) Acknowledge task completion |
| `/internal/heartbeat` | POST | (Worker) Report worker heartbeat |
| `/health` | GET | Health check |
| `/metrics` | GET | Prometheus metrics |

## Data Models

Request and response bodies are Pydantic models from `src/jobqueue/models.py`.

### Task

Returned by `POST /tasks`, `GET /tasks/{task_id}`, and `POST /tasks/{task_id}/retry`.

```json
{
  "id": "3f1c...uuid",
  "name": "send_email",
  "queue": "default",
  "payload": {"to": "user@example.com"},
  "priority": 2,
  "retries": 0,
  "max_retries": 3,
  "timeout_ms": 30000,
  "eta": null,
  "created_at": "2026-07-01T10:30:00Z",
  "started_at": null,
  "completed_at": null,
  "idempotency_key": null,
  "metadata": {},
  "status": "pending",
  "worker_id": null
}
```

- `id` — UUID assigned by the server.
- `priority` — `TaskPriority` enum: `CRITICAL=0`, `HIGH=1`, `NORMAL=2` (default), `LOW=3`.
- `status` — `TaskStatus` enum: `pending`, `queued`, `running`, `success`, `failure`, `retry`, `cancelled`.
- `timeout_ms` — task timeout in **milliseconds** (default `30000`).

### TaskCreate (request body for `POST /tasks`)

```json
{
  "name": "send_email",
  "queue": "default",
  "payload": {"to": "user@example.com"},
  "priority": 2,
  "max_retries": 3,
  "timeout_ms": 30000,
  "eta": null,
  "idempotency_key": null,
  "metadata": {}
}
```

Only `name` is required. All other fields have the defaults shown above.

### TaskResult

Returned by `GET /tasks/{task_id}/result` and used as the request body for
`POST /internal/acknowledge`.

```json
{
  "task_id": "3f1c...uuid",
  "status": "success",
  "result": {"message_id": "msg-123"},
  "error": null,
  "traceback": null,
  "started_at": "2026-07-01T10:30:30Z",
  "completed_at": "2026-07-01T10:31:00Z",
  "duration_ms": 30000.0,
  "worker_id": "worker-abc123"
}
```

### QueueStats

Returned by `GET /queues/{queue_name}/stats`.

```json
{
  "name": "default",
  "pending": 45,
  "running": 10,
  "completed": 1523,
  "failed": 12,
  "total": 1590
}
```

## Endpoints

### Task Management

#### Create Task

**`POST /tasks`** — Create and enqueue a new task. Response model: `Task`.

**Request body:** a `TaskCreate` object (see above).

**Response:** the enqueued `Task` (with a server-assigned `id` and `status`).

**Example:**
```bash
curl -X POST http://localhost:8000/tasks \
  -H "Content-Type: application/json" \
  -d '{"name": "send_email", "payload": {"to": "user@example.com"}, "priority": 1}'
```

#### Get Task

**`GET /tasks/{task_id}`** — Get a task by ID. Response model: `Task`.

- `200` — the `Task`.
- `404` — task not found.

```bash
curl http://localhost:8000/tasks/<task_id>
```

#### Get Task Result

**`GET /tasks/{task_id}/result`** — Get the result of a completed task.
Response model: `TaskResult`.

- `200` — the `TaskResult`.
- `202` — task exists but is not finished yet (status is not `success`/`failure`);
  the body's `detail` reports the current status.
- `404` — task not found, or the task finished but no result was stored.

```bash
curl http://localhost:8000/tasks/<task_id>/result
```

#### Cancel Task

**`DELETE /tasks/{task_id}`** — Cancel a pending task.

- `200` — `{"status": "cancelled", "task_id": "<task_id>"}`.
- `400` — the task cannot be cancelled (not found or already processed).

```bash
curl -X DELETE http://localhost:8000/tasks/<task_id>
```

#### Retry Task

**`POST /tasks/{task_id}/retry`** — Manually requeue a failed or cancelled task.
Response model: `Task`.

Takes no request body. Only tasks with status `failure` or `cancelled` can be retried.

- `200` — the requeued `Task`.
- `400` — task is not in a retryable state (`detail` reports the current status).
- `404` — task not found.

```bash
curl -X POST http://localhost:8000/tasks/<task_id>/retry
```

### Queue Management

#### List Queues

**`GET /queues`** — List all known queues.

**Response:** `{"queues": ["default", "priority", ...]}`. Returns an empty list
if the broker does not support queue enumeration.

```bash
curl http://localhost:8000/queues
```

#### Get Queue Statistics

**`GET /queues/{queue_name}/stats`** — Get statistics for a single queue.
Response model: `QueueStats` (see above).

```bash
curl http://localhost:8000/queues/default/stats
```

### Worker Endpoints (Internal)

These routes are used by workers to communicate with the broker. They are
protected by API-key auth when `API_KEYS` is set.

#### Dequeue

**`POST /internal/dequeue`** — Dequeue a task for processing.

**Query parameters:**
- `queues` (required, repeatable) — queue names to poll, e.g. `?queues=default&queues=priority`.
- `timeout` (optional) — long-poll timeout in seconds, `0`–`30` (default `0`).

**Response:** `{"task": {...}}` with the `Task` payload, or `{"task": null}` if
no task was available.

```bash
curl -X POST "http://localhost:8000/internal/dequeue?queues=default&timeout=5"
```

#### Acknowledge

**`POST /internal/acknowledge`** — Acknowledge completion of a task.

**Request body:** a `TaskResult` object (see above).

**Response:** `{"status": "acknowledged"}`.

#### Heartbeat

**`POST /internal/heartbeat`** — Report a worker heartbeat.

**Query parameters:**
- `worker_id` (required) — the worker's identifier.
- `current_task` (optional) — the ID of the task the worker is currently running.

**Response:** `{"status": "ok"}`.

```bash
curl -X POST "http://localhost:8000/internal/heartbeat?worker_id=worker-1&current_task=<task_id>"
```

### Monitoring

#### Health Check

**`GET /health`** — Liveness check. Always open (no auth).

**Response:**
```json
{
  "status": "healthy",
  "timestamp": "2026-07-01T10:50:00Z",
  "version": "0.1.0"
}
```

#### Metrics

**`GET /metrics`** — Prometheus metrics in the standard text exposition format.
Always open (no auth). The app exports:

- `jobqueue_tasks_enqueued_total{queue, priority}` (counter)
- `jobqueue_tasks_completed_total{queue, status}` (counter)
- `jobqueue_task_duration_seconds{queue}` (histogram)

```bash
curl http://localhost:8000/metrics
```

## Error Responses

Errors raised by route handlers use FastAPI's default shape:

```json
{"detail": "Task not found"}
```

Errors from the hardening middleware (auth, rate limiting, timeout) use the same
`{"detail": "..."}` shape with status codes `401`, `429`, and `504` respectively
(see [Authentication](#authentication)). Request-body validation failures return
FastAPI's standard `422` response with a `detail` array describing the invalid
fields.
